# 코딩 규칙

본 레포는 한국공학대학교 소프트웨어공학 (01) 4조 학기 프로젝트의 일부이다.
학술 평가와 팀원 간 코드 리뷰가 주된 사용처임을 전제로 한다.

## 1. 곁가지 기능을 만들지 않는다

- 유스케이스 명세에 있는 기능만 구현한다.
- "혹시 나중에 쓸지도" 하는 추상화, 옵션, 플래그를 미리 두지 않는다.
- 프레임워크·라이브러리가 이미 제공하는 기능을 직접 다시 만들지 않는다.
- 기존 기능을 고칠 때, 주문받지 않은 리팩토링을 함께 끼워 넣지 않는다.

## 2. 학부생 수준에서 이해 가능해야 한다

- 메타프로그래밍, 과한 제너릭, 데코레이터 체인 등은 피한다.
- 한 함수의 책임이 한눈에 들어와야 한다. 분기·중첩이 깊으면 분리한다.
- 익숙하지 않은 패턴(예: CQRS, 이벤트 소싱, DI 컨테이너)은 도입 전 팀과 합의한다.
- 3줄 비슷한 코드가 보인다고 바로 추상화하지 않는다. 같은 패턴이 세 번
  반복되고, 변하는 축이 명확해진 뒤에 묶는다.

## 3. 코드는 자기 설명적이어야 한다

- 이름이 곧 설명이다. `x`, `tmp`, `do_it`, `data2` 같은 모호한 이름 금지.
- 주석은 *왜*를 적는다. *무엇을*은 코드가 이미 보여준다.
- 주석에 이모지를 넣지 않는다. 발표 자료·커밋 메시지·PR 본문에서도 동일.
- 주석으로 죽은 코드를 남기지 않는다. 필요하면 git 히스토리에서 찾는다.

## 4. 학술 자료로서의 일관성

- 명세서 / 다이어그램 / 코드의 용어가 같아야 한다.
  ("예약" vs "Reservation" vs "booking" 혼용 금지.)
- 도메인 용어는 한 곳에 정의(README 또는 설계 문서)하고 코드는 그것을 따른다.
- 한국어 주석을 기본으로 한다. 영어 식별자(클래스·함수명)와 섞어 써도 좋다.

## 5. 변경 절차

- 디렉토리 구조를 바꾸려면 `tree.md` 를 먼저 갱신한다.
- 외부에 노출되는 계약(API 응답, 메트릭 JSON 스키마 등)을 바꾸려면
  설계 문서를 먼저 갱신하고 코드를 따른다.
